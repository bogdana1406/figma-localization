# Figma Localization

Система перевода Figma-презентаций: выгружает тексты из макета, переводит через
Gemini, вычитывается вручную и вписывает переводы обратно в Figma. Умеет работать
инкрементально — при обновлении презентации переводит только новое, не трогая уже
проверенное.

Состоит из двух частей:
- **Python-скрипты** — выгрузка, перевод, сборка файлов (запускаются в терминале).
- **Figma-плагин** — вписывает готовые переводы обратно в макет.

## Как это устроено

Ключевых понятия три, и они независимы друг от друга:

**Экспорт** — снимок текстов английского макета на момент времени. Лежит в
`projects/<проект>/exports/` с датой в имени. Каждый новый экспорт — новый снимок.

**Память переводов** (`memory.csv`) — накопленный архив «английский текст → перевод»
для одного языка. Не зависит от версии макета и от id узлов: переживает любые
пересоздания файла. Наполняется только **вычитанными** (утверждёнными) переводами.

**id узлов** — уникальный идентификатор узла внутри одного файла Figma. Используется
плагином, чтобы точно вписать перевод. При копировании файла целиком (Duplicate) id
сохраняются; при копировании отдельных слайдов между файлами — меняются.

Диффы («что нового в макете») считаются между двумя **экспортами** — это чисто
англоязычная операция, не зависящая от языка перевода. Память подключается уже
на этапе сборки конкретного языка.

## Установка

Нужен Python 3 и десктопное приложение Figma (для плагина).

```bash
# 1. Клонировать/открыть папку проекта
cd figma-localization

# 2. Создать виртуальное окружение (один раз) — подробности в VENV.md
python3 -m venv .venv
source .venv/bin/activate

# 3. Поставить зависимости
pip install -r requirements.txt
```

Создать файл `.env` в корне проекта с ключами:

```
FIGMA_TOKEN=figd_...        # Figma → Settings → Security → Personal access tokens
GCP_API_KEY=...             # ключ Google Gemini API
GEMINI_MODEL=gemini-3.5-flash   # опционально: переопределить модель перевода
```

Токену Figma достаточно прав **File content: read**.

## Настройка проекта

Проекты описываются в `projects.json`. Каждая презентация — отдельный блок:

```json
{
  "hyperocket": {
    "file_key": "Kx1AG5yGnmVfwhppPj9AMG",
    "file_url": "https://www.figma.com/design/Kx1AG5yGnmVfwhppPj9AMG/HypeRocket---en",
    "pages": ["Presentation"]
  }
}
```

- `file_key` — часть URL файла Figma между `/design/` и названием.
- `pages` — какие страницы выгружать (по имени). `["*"]` или пропустить — все страницы.

Промпты для перевода лежат в `projects/<проект>/prompts/<язык>.txt` — своя настройка
на каждую пару проект+язык (там задаётся, что не переводить: бренды, термины и т.п.).

Когда приходит новая версия презентации от дизайнера — обновляете `file_key` в
`projects.json` на новый. История версий сохраняется в `exports/`.

## Полный цикл: перевод с нуля

Когда презентация переводится впервые.

```bash
# 1. Выгрузить тексты из Figma
python export_figma.py hyperocket

# 2. Подготовить к переводу (дедупликация; памяти ещё нет — все узлы новые)
python prepare.py hyperocket portuguese

# 3. Перевести через Gemini
python translate.py hyperocket portuguese

# 4. ВЫЧИТКА вручную:
#    открыть projects/hyperocket/translations/portuguese/to_translate.csv
#    поправить переводы в колонке figma_text, сохранить

# 5. Собрать финальный файл + наполнить память
python finalize.py hyperocket portuguese

# 6. Файл для дизайнера (для сверки)
python make_design.py hyperocket portuguese

# 7. Импортировать в Figma — через плагин (см. ниже)
```

Первый раз перед `finalize` память создаётся автоматически из вычитки. Если у вас
уже есть **готовый вычитанный** перевод (из старого процесса), положите его в
`projects/hyperocket/translations/portuguese/hyperocket-portuguese.csv` и соберите
из него память один раз:

```bash
python build_memory.py hyperocket portuguese
```

## Инкрементальный цикл: обновление презентации

Когда презентация уже переведена, а дизайнер прислал новую версию (добавил слайды,
поправил тексты). Переводится и вычитывается **только новое** — старое берётся из памяти.

```bash
# 1. Обновить file_key в projects.json на новую версию, затем выгрузить
python export_figma.py hyperocket
#    (в exports/ теперь два снимка: старый и новый)

# 2. prepare сравнит новый экспорт со старым, отсеет то, что уже в памяти,
#    и оставит на перевод только новое/изменённое
python prepare.py hyperocket portuguese
#    Показывает: без изменений / изменённые / новые / удалённые
#    to_translate.csv будет содержать только то, что реально надо перевести

# 3. Перевести только новое
python translate.py hyperocket portuguese

# 4. ВЫЧИТКА нового (маленький файл, а не вся презентация)

# 5. Собрать финал: старое из памяти + свежевычитанное новое
python finalize.py hyperocket portuguese

# 6. Файл для дизайнера
python make_design.py hyperocket portuguese

# 7. Импорт через плагин
```

`prepare` по умолчанию берёт **два самых свежих экспорта по дате** и показывает
выбранную пару перед работой. Можно указать явно:

```bash
python prepare.py hyperocket portuguese --old путь/к/старому.csv --new путь/к/новому.csv
```

## Плагин: импорт перевода в Figma

### Подключение (один раз)

Только в **десктопном** приложении Figma:
Plugins → Development → «Import plugin from manifest…» → выбрать `plugin/manifest.json`.

### Использование

1. **Обязательно работайте на дубликате файла** — плагин пишет в живой макет.
2. Запустить плагин, загрузить `hyperocket-<язык>.csv` (финальный файл из `finalize`).
3. Выбрать режим, нажать **Предпросмотр** (ничего не пишет, только показывает план).
4. Проверить числа и список слайдов, затем **Применить**.

После записи плагин **выделяет** записанные узлы (видно, что изменилось) и предлагает
**скачать отчёт** для дизайнера (`...-imported-<дата>.csv`).

### Два режима поиска узлов

**По id** — когда работаете со свежим дубликатом того же файла, из которого делали
экспорт. Находит узлы точно по id. Поддерживает фильтр **«только новые и изменённые»**
(через колонку category) — старые узлы не трогает, бережёт правки дизайнера.
Требует, чтобы id в файле совпадали с id в CSV.

**По тексту** — когда в переведённый файл добавили новые английские слайды (id у них
чужие). Находит узлы по английскому тексту, вписывает перевод во все вхождения. Уже
переведённые (не английские) узлы не трогает. Область работы ограничивается
**выделением** (безопасно), текущей страницей или всем файлом.
В список «не найдено» попадает английский текст, которого нет в CSV — сигнал, что его
надо доперевести.

### Опции

- **Только новые и изменённые** (режим по id) — не трогать unchanged-узлы.
- **Переводить имена слоёв** — по умолчанию выкл (имена слоёв остаются английскими).
  Вкл — имена едут за переводом (как делал CopyDoc).

### Проверка «тот ли файл»

Если из CSV нашлась лишь часть узлов — вероятно, открыт файл не той версии (id не
совпадают). Плагин покажет число найденных; малое число = не тот файл или не тот режим.

## Флоу с дизайнером

Типичный сценарий обновления, бережный к работе дизайнера:

1. Дизайнер дорабатывает **английскую** версию, присылает.
2. Вы делаете дубликат, обновляете `file_key`, гоните инкрементальный цикл.
3. Импортируете **по id, «только новые»** в дубликат — старые слайды остаются
   английскими, новые/изменённые переводятся.
4. Дизайнер получает два файла в Google Таблицах:
   - `...-design-<дата>.csv` — весь перевод для сверки (фильтр по category = что новое).
   - `...-imported-<дата>.csv` — что именно вписано и на каких слайдах.
5. Дизайнер проверяет переведённые слайды, правит стили (если перевод не влез),
   и копирует готовые слайды в свою уже оформленную переведённую презентацию.

Так её старые правки оформления не затираются — добавляются только новые слайды.

## Структура файлов

```
figma-localization/
├── .env                          # токены (не в git)
├── projects.json                 # конфиг проектов
├── requirements.txt
├── VENV.md                       # шпаргалка по виртуальному окружению
├── README.md
│
├── export_figma.py               # выгрузка узлов из Figma (REST API)
├── build_memory.py               # сборка памяти из готового перевода (разово)
├── prepare.py                    # дифф + отсев по памяти + дедупликация
├── translate.py                  # перевод через Gemini
├── finalize.py                   # сборка финала + обновление памяти
├── make_design.py                # файл для дизайнера
│
├── plugin/                       # Figma-плагин (импорт в макет)
│   ├── manifest.json
│   ├── code.js
│   └── ui.html
│
└── projects/
    └── hyperocket/
        ├── prompts/              # промпты по языкам (в git)
        │   └── portuguese.txt
        ├── exports/              # снимки макета (не в git)
        │   └── 2026-07-09_1154_export.csv
        └── translations/
            └── portuguese/       # рабочие файлы языка (не в git)
                ├── memory.csv            # память переводов (архив)
                ├── to_translate.csv      # на перевод и вычитку
                ├── layout.csv            # карта раскладки (служебная)
                ├── hyperocket-portuguese.csv          # финал
                └── hyperocket-portuguese-design-*.csv # для дизайнера
```

В git попадают только код, конфиг и промпты. Данные (экспорты, переводы, память)
игнорируются — см. `.gitignore`.

## Частые вопросы и проблемы

**`externally-managed-environment` при pip** — вы вне виртуального окружения.
Активируйте: `source .venv/bin/activate`. Подробности в `VENV.md`.

**`ModuleNotFoundError`** — не активировано окружение или не поставлены зависимости.
Проверьте `which python` (должен вести в `.venv`), затем `pip install -r requirements.txt`.

**Gemini: «model is no longer available» (404)** — Google отключил модель. Смените
её через `.env`: `GEMINI_MODEL=<новая-модель>`. Актуальные модели — на
`ai.google.dev/gemini-api/docs/models`. Скрипт при повторном запуске допереведёт
пропущенное.

**Плагин: много «не найдено» в режиме по id** — открыт файл не той версии, чем тот,
из которого собран CSV (id не совпадают). Работайте с дубликатом того же файла.

**Плагин не виден в Figma** — плагины из manifest подхватывает только десктопное
приложение, не браузер.

**Числа в предпросмотре нужно перепроверить** — предпросмотр ничего не пишет, можно
жать сколько угодно. На одних данных числа стабильны.

**Перед импортом всегда дублируйте файл** — плагин пишет в живой макет, дубликат
это страховка.
